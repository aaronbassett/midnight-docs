# Deploy with a maintenance authority and re-join

Deploying with an authority is the step that makes everything else in this tutorial possible. A contract deployed without a `signingKey` is permanently non-upgradable, so this part deploys version one with one, then re-joins the deployment to confirm the maintenance interfaces are there.

## Prerequisites

- [The wallet and providers](./project-setup.mdx) are in place and `npm install` has run.
- Both contract versions are compiled with the full ZK pipeline.
- Your `.env` has a funded `MN_UPGRADE_WALLET_MNEMONIC` on the target network.

## Deploy version one

`src/deploy-v1.ts` deploys the version one contract. If `MN_UPGRADE_SIGNING_KEY` is unset it samples a fresh signing key, derives the verifying key, and passes the signing key to `deployContract`.

```typescript title="src/deploy-v1.ts"
const env = loadUpgradeEnv();
const config = resolveConfig(env);
const signingKey = env.signingKey ?? sampleSigningKey();
const verifyingKey = signatureVerifyingKey(signingKey);

// ... build the wallet, wait for sync and funds, create providers ...

const deployed = await deployContract(providers as never, {
  compiledContract,
  privateStateId: 'upgradableCounterPrivateState',
  initialPrivateState: {},
  signingKey,
  args: [env.initialCounter, env.initialStep],
} as never);
```

The `signingKey` field is the whole point. Passing it sets the deploying key as the contract's maintenance authority, which is what enables updatability. Omit it and the contract deploys with the default empty committee and threshold `1`, a condition nothing can satisfy, and no verifier key can ever be changed. The `args` seed the constructor with the initial counter and step from your environment. See [`DeployContractOptions`](/api-reference/midnight-js/@midnight-ntwrk/midnight-js-contracts/type-aliases/DeployContractOptions) for the full option set.

After the deploy lands, the script writes the manifest with the contract address and the maintenance verifying key, then runs it:

```bash
npm run deploy:v1
```

When the script samples a key, it prints it once:

```text
[upgrade] network=preprod
[upgrade] deploying version=v1
[upgrade] using maintenance verifying key=0x...
[upgrade] sampled a fresh maintenance signing key for this deployment
[upgrade] store this signing key securely; it is required for later maintenance actions
[upgrade] maintenance signing key=0x...
[upgrade] deployed contract=0200...
[upgrade] wrote non-secret manifest to /path/to/data/latest-deployment.json
```

:::warning
The maintenance signing key is printed exactly once, at deploy time, and is never written to the manifest. Copy it into `MN_UPGRADE_SIGNING_KEY` in `.env` now. Whoever holds this key controls every future update and every authority change on this contract. Lose it and the contract is frozen on its current version for good.
:::

Record two values from this run before moving on:

- `MN_UPGRADE_SIGNING_KEY`, the printed signing key.
- `MN_UPGRADE_CONTRACT_ADDRESS`, the deployed contract address.

## Re-join and inspect

`src/inspect-deployment.ts` re-opens the deployment at a known address and reads its maintenance interfaces. This is the same join pattern every maintenance operation uses, so it is worth confirming it works before you rely on it.

```typescript title="src/inspect-deployment.ts"
const found = await findDeployedContract(providers as never, {
  contractAddress: assertContractAddress(env.contractAddress),
  compiledContract,
  privateStateId: 'upgradableCounterPrivateState',
  initialPrivateState: {},
  signingKey: env.signingKey,
} as never);

const state = await providers.publicDataProvider.queryContractState(env.contractAddress);
const ledgerState = state ? contractModule.ledger(state.data) : null;

console.log(`[upgrade] joined contract=${found.deployTxData.public.contractAddress}`);
console.log(
  `[upgrade] maintenance interfaces increment=${typeof found.circuitMaintenanceTx.increment} ` +
    `replaceAuthority=${typeof found.contractMaintenanceTx.replaceAuthority}`,
);
console.log(
  `[upgrade] ledger counter=${ledgerState?.counter ?? 'n/a'} step=${ledgerState?.step ?? 'n/a'}`,
);
```

`findDeployedContract` re-opens the same address locally without redeploying. The object it returns exposes the two maintenance surfaces you use later: [`circuitMaintenanceTx`](/api-reference/midnight-js/@midnight-ntwrk/midnight-js-contracts/type-aliases/FoundContract#circuitmaintenancetx), which carries one interface per circuit for inserting and removing verifier keys, and [`contractMaintenanceTx`](/api-reference/midnight-js/@midnight-ntwrk/midnight-js-contracts/type-aliases/FoundContract#contractmaintenancetx), which carries `replaceAuthority`. Reading the ledger through `contractModule.ledger` decodes the public state into the `counter` and `step` you deployed.

Run it against your deployment. The inspect script needs the contract address and the version to load.

```bash
export MN_UPGRADE_CONTRACT_ADDRESS="<deployed address>"
export MN_UPGRADE_CONTRACT_VERSION=v1
npm run inspect
```

```text
[upgrade] joined contract=0200...
[upgrade] compiled view=v1
[upgrade] maintenance interfaces increment=object replaceAuthority=function
[upgrade] ledger counter=0 step=1
```

The `increment=object replaceAuthority=function` line confirms the maintenance interfaces are live on the found contract. The ledger values match what you deployed.
